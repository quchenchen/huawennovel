# Proofread Skill — 中文文稿审校

日常单篇稿件的系统化审校工具，覆盖错字语病、信息核查、政治宗教表述三层检查。

---

## Skill 触发条件

当用户提到以下任一场景时激活本 Skill：
- "审校" / "校对" / "审一下" / "校一下"
- "检查稿件" / "帮我看看这篇稿子"
- "查错别字" / "查语病" / "检查文字"
- "核查信息" / "查一下事实" / "核实数据"
- "检查敏感表述" / "政治检查" / "宗教表述检查"
- 将稿件文件放入 `proofread/inbox/` 目录

---

## 功能模块

### 1. 快速审校（默认流程）

**触发**: "审校这篇稿子" / "帮我审一下" / 文件放入 inbox

**审校范围**: 三层全做（错字语病 + 信息核查 + 政治宗教）

---

**步骤 1：接收稿件 + 识别类型**

```bash
# 稿件来源：
# A. 用户指定文件路径 → read_file 读取
# B. 用户直接粘贴文本 → 直接使用
# C. 用户放入 proofread/inbox/ → 扫描新文件

# 扫描 inbox
ls -lt proofread/inbox/
```

**识别稿件类型**：

| 类型 | 严格度 | 说明 |
|------|--------|------|
| 新闻稿 | ⭐⭐⭐⭐⭐ | 事实准确性要求最高 |
| 公文/红头文件 | ⭐⭐⭐⭐⭐ | 格式、用语规范要求最高 |
| 学术论文 | ⭐⭐⭐⭐ | 数据、引用准确性 |
| 公众号/自媒体 | ⭐⭐⭐ | 可适当放宽，注重可读性 |
| 博客/随笔 | ⭐⭐ | 侧重错字语病 |
| 小说/文学 | ⭐⭐ | 侧重文学性，政治检查适度 |

**确认审校范围**：
- 默认三层全做
- 用户可指定："只查错别字" / "只做政治检查" / "不做信息核查"

**读取参考规则库**：
```bash
# 必须读取的规则文件
read_file proofread/references/common-errors.md       # 常见错字语病例库
read_file proofread/references/terminology-standard.md # 规范用语对照表
read_file proofread/references/political-sensitive.md  # 政治敏感词库
read_file proofread/references/religious-sensitive.md  # 宗教敏感表述库

# 如果用户提供了背景文档（如参考资料、数据来源），也一并读取
```

---

**步骤 2：错字语病检查**

逐段逐句扫描，按以下类别分类输出。

#### 2.1 错别字

检查范围：
- 形近字混淆（帐/账、的/地/得、做/作）
- 音近字混淆（必须/必需、反映/反应）
- 繁简混用
- 拼音输入导致的同音错误

输出格式：
```
### 错别字（X处）
| # | 位置 | 原文 | 建议 | 类型 |
|---|------|------|------|------|
| 1 | 第3段第2句 | "帐号" | "账号" | 形近字 |
```

#### 2.2 语法语病

检查范围：
- 成分残缺（缺主语、缺谓语、缺宾语）
- 搭配不当（动宾、偏正、主谓）
- 语序不当
- 句式杂糅（"原因是……导致的"）
- 重复赘余（"大约……左右"、"亲眼目睹"）
- 关联词搭配错误
- 否定不当（多重否定导致意思反转）
- 代词指代不明

输出格式：
```
### 语法语病（X处）
| # | 位置 | 问题 | 类型 | 修改建议 |
|---|------|------|------|---------|
| 1 | 第2段 | 主语缺失 | 成分残缺 | "通过这次活动，使大家……" → 删"通过"或"使" |
```

#### 2.3 标点符号

检查范围：
- 逗号滥用（一逗到底）
- 句号/问号/叹号误用
- 引号、括号、书名号使用错误
- 省略号规范（六个点"……"，非三个点"..."）
- 破折号规范（两个"——"，非一个"—"）
- 顿号与逗号混用
- 标点与文字之间多余空格

输出格式：
```
### 标点符号（X处）
| # | 位置 | 原文 | 建议 | 规则 |
|---|------|------|------|------|
| 1 | 第1段 | 逗号连续5个未断句 | 拆分为多个短句 | 避免一逗到底 |
```

#### 2.4 用词不当

检查范围：
- 口语/书面语混用（稿件类型不匹配）
- 词语色彩不当（褒贬误用）
- 生造词语
- 外来词使用不规范
- 成语误用（望文生义、对象误用、谦敬误用）
- 极端评价词（"最佳""最好""最著名""最先进"等）

输出格式：
```
### 用词不当（X处）
| # | 位置 | 原词 | 建议 | 原因 |
|---|------|------|------|------|
| 1 | 第3段 | "立马" | "立即" | 新闻稿应使用书面语 |
| 2 | 第5段 | "罄竹难书"形容好事 | 删除或替换 | 成语褒贬误用 |
```

#### 2.5 格式一致性

检查范围：
- 中英文之间是否加空格（正式稿件建议加）
- 数字格式统一（阿拉伯数字 vs 中文数字）
- 引号全半角统一（中文语境使用 "" "" '' ''）
- 括号全半角统一（中文语境使用（））
- 列表格式统一
- 标题层级统一
- 人名/地名/机构名全文一致性

输出格式：
```
### 格式一致性（X处）
| # | 问题 | 出现位置 | 说明 |
|---|------|---------|------|
| 1 | 数字格式不统一 | 全文 | "2024年"与"二〇二四年"混用 |
| 2 | 机构名称不一致 | 第2段/第6段 | "中国银行"vs"中行" |
```

---

**步骤 3：信息核查**

将全文中的事实性陈述、数据、专有名词提取出来逐一核查。

#### 3.1 事实性信息核查

提取并检查：
- 时间/日期（事件发生时间、历史日期）
- 人名（职务、拼写、用字）
- 地名（行政区划变更、规范名称）
- 机构名（全称 vs 简称，是否更名）
- 数据/统计数字
- 法律法规名称及条文编号
- 历史事件描述
- 科学常识

输出格式：
```
### 事实性信息核查

#### ✅ 已核实无误（X处）
| # | 内容 | 核查结果 | 依据 |
|---|------|---------|------|
| 1 | "中华人民共和国成立于1949年" | 正确 | — |

#### ❌ 发现错误（X处）
| # | 位置 | 原文 | 正确信息 | 依据 |
|---|------|------|---------|------|
| 1 | 第3段 | "成立于1998年" | 成立于1999年 | [来源说明] |

#### ⚠️ 存疑待确认（X处）
| # | 位置 | 内容 | 疑问 | 建议 |
|---|------|------|------|------|
| 1 | 第4段 | "据内部人士透露" | 无来源标注 | 补充信源或删除 |
| 2 | 第6段 | "增长30%" | 无数据来源 | 添加出处注释 |
```

> **核查方式**：AI 基于已有知识库进行交叉验证。对于高置信度的信息直接标注"已核实"，对于不确定的信息标注"存疑待确认"。如需联网搜索，向用户说明并请求授权。

#### 3.2 引文核查

检查范围：
- 直接引用的文字是否准确
- 引用来源是否正确标注
- 引用是否断章取义
- 数据引用是否标注来源

输出格式：
```
### 引文核查（X处）
| # | 位置 | 引文内容 | 问题 | 建议 |
|---|------|---------|------|------|
| 1 | 第3段 | "引用XX讲话" | 未标注出处 | 补充来源 |
| 2 | 第5段 | "据XX报告" | 数据与原始报告不符 | 核实后修正 |
```

#### 3.3 逻辑一致性检查

检查范围：
- 前后数据是否矛盾
- 时间线是否自洽
- 因果关系是否成立
- 概念定义是否前后一致

输出格式：
```
### 逻辑一致性（X处）
| # | 位置 | 问题 | 说明 |
|---|------|------|------|
| 1 | 第2段 vs 第5段 | 数据矛盾 | 前文"增长10%"，后文"增长15%" |
| 2 | 第4段 vs 第7段 | 时间线冲突 | 事件A在事件B之后发生，但描述顺序相反 |
```

**步骤 4：政治宗教表述检查**

⚠️ **本步骤为必检项，不可跳过。** 即使稿件类型为小说/随笔，仍需执行基本检查。

#### 4.1 政治敏感表述（高风险）

检查范围（参考 `references/political-sensitive.md`）：
- 国家主权与领土完整相关表述
- 台湾、香港、澳门相关表述
- 涉藏、涉疆相关表述
- 党和国家领导人姓名、职务表述
- 党的历史事件表述
- 涉外表述（国家排序、国际组织名称）
- 涉军表述
- 民族关系表述
- 涉密信息
- 新闻报道中的禁用词和慎用词（新华社规范）

输出格式：
```
### 🚨 政治敏感表述（X处）

#### ❌ 必须修改（X处）
| # | 位置 | 原文 | 问题 | 修改建议 |
|---|------|------|------|---------|
| 1 | 第3段 | "台湾国" | ❌ 违反一个中国原则 | "中国台湾地区" |

#### ⚠️ 建议修改（X处）
| # | 位置 | 原文 | 问题 | 修改建议 |
|---|------|------|------|---------|
| 1 | 第2段 | 国家列举顺序 | 按惯例排序 | 参照外交部网站 |
```

#### 4.2 宗教敏感表述

检查范围（参考 `references/religious-sensitive.md`）：
- 宗教名称、教义表述是否准确
- 是否对特定宗教有不尊重表述
- 宗教与极端主义的关联是否过度简化
- 宗教节日、仪式描述是否准确
- "邪教"认定是否有法律依据
- 民族风俗与宗教的区分是否正确
- 涉穆斯林表述规范
- 清真概念使用规范

输出格式：
```
### 宗教表述检查（X处）
| # | 位置 | 原文 | 问题 | 建议 |
|---|------|------|------|------|
| 1 | 第6段 | 对某宗教仪式描述不准确 | 事实性错误 | 建议咨询专业人士核实 |
| 2 | 第8段 | 将"某宗教"与恐怖主义直接关联 | 标签化/过度简化 | 区分宗教信仰与极端行为 |
```

#### 4.3 规范用语检查

检查范围（参考 `references/terminology-standard.md`）：
- 新华社规范用语
- 国家标准术语
- 法律法规标准用语
- 行业规范用语
- 已更改的旧称/旧译
- 禁用不文明用语

输出格式：
```
### 规范用语（X处）
| # | 位置 | 原文 | 规范表述 | 依据 |
|---|------|------|---------|------|
| 1 | 第1段 | "非典" | "传染性非典型肺炎" | 新华社规范 |
| 2 | 第4段 | "新冠肺炎" | "新型冠状病毒感染" | 2022年12月后规范 |
| 3 | 第3段 | "残疾人" | "残障人士" | 当代规范用语 |
```

#### 4.4 用语倾向性检查

检查范围：
- 是否对特定群体存在偏见性表述
- 是否存在性别歧视用语
- 是否存在地域歧视用语
- 是否存在职业歧视用语
- 褒贬词使用是否得当

输出格式：
```
### 用语倾向性（X处）
| # | 位置 | 原文 | 问题 | 建议 |
|---|------|------|------|------|
| 1 | 第5段 | "农民工涌入城市" | "涌入"带有负面色彩 | "农民工进入城市" |
| 2 | 第7段 | "女司机" | 性别标签化 | 直接说"司机" |
```

**步骤 6：执行修改**

#### 选项 A：直接修改稿件

```bash
# 1. 备份原稿
cp [原稿路径] proofread/inbox/[原稿名].bak

# 2. 按审校结果修改稿件
# 3. 生成修订版（修改处用删除线+加粗标注）

# 4. 输出修改对比
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📝 修改对比
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

| 位置 | 修改前 | 修改后 |
|------|--------|--------|
| 第3段 | ~~帐号~~ | **账号** |
| 第5段 | ~~非典~~ | **传染性非典型肺炎** |

共修改 X处，新增 X字，删除 X字。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 选项 B：导出审校报告

```bash
# 生成报告文件
# proofread/reports/YYYY-MM-DD-[稿件标题].md
```

报告包含完整的四步检查结果 + 汇总，格式即步骤 2-5 的输出内容。

#### 选项 C：原文批注

在原文对应位置插入批注标记：
```markdown
原文内容<!-- ⚠️ 审校批注：[问题类型] 原文"XX"建议改为"YY" -->继续原文
```

#### 选项 D：只处理高风险

仅修改 🔴 高风险项，其余跳过。

---

**步骤 7：存档 + 记录**

审校完成后执行：

```bash
# 1. 生成报告到 reports/
# 2. 更新规则库（发现新的常见错误、新的敏感表述时）
# 3. Git 提交（如果启用了版本管理）
```

在 `proofread/reports/` 中保存报告，同时更新审校日志。

### 2. 单项检查

**触发**: 用户明确指定只做某一项检查

**支持的单项**：

| 指令 | 执行内容 |
|------|---------|
| "只查错别字" | 仅执行步骤 2（错字语病检查） |
| "只做信息核查" | 仅执行步骤 3（信息核查） |
| "只做政治检查" | 仅执行步骤 4（政治宗教表述检查） |
| "查一下标点" | 仅执行步骤 2.3（标点符号） |
| "查一下格式" | 仅执行步骤 2.5（格式一致性） |
| "查一下引文" | 仅执行步骤 3.2（引文核查） |

单项检查完成后直接输出对应模块的报告，跳过其他步骤。

### 3. 批量审校

**触发**: "批量审校" / 文件放入 `proofread/inbox/`

**操作流程**：


### 4. 管理规则库

**触发**: "更新规则库" / "添加敏感词" / "更新错字库"

#### 4.1 添加政治敏感词

用户发现新的敏感表述时，追加到 `references/political-sensitive.md`：

```markdown
### [分类名称]
- **错误表述**: "XXX"
- **正确表述**: "YYY"
- **说明**: [为什么不能用错误表述]
- **添加日期**: YYYY-MM-DD
- **来源**: [用户反馈/AI发现/官方文件]
```

#### 4.2 添加常见错字

```markdown
### [分类：形近字/音近字/搭配]
- **错误**: "XXX"
- **正确**: "YYY"
- **示例**: "XX年XX月，XXX"
- **添加日期**: YYYY-MM-DD
```

#### 4.3 添加规范用语

```markdown
### [领域]
- **旧称/不规范**: "XXX"
- **规范表述**: "YYY"
- **依据**: [来源文件/标准]
- **生效日期**: YYYY-MM-DD
- **添加日期**: YYYY-MM-DD
```

#### 4.4 更新流程

```bash
# 1. 读取对应规则文件
# 2. 确认新内容不重复
# 3. 追加到对应分类
# 4. Git 提交
git add proofread/references/
git commit -m "更新规则库: [新增/更新] XX条[分类名]"
```

## 项目结构

```
proofread/
├── inbox/                              # 待审稿件
│   └── README.md                       # 使用说明
├── done/                               # 已审稿件（处理后移入）

│   └── YYYY-MM-DD-[稿件标题].md
├── references/                         # 参考规则库
│   ├── political-sensitive.md          # 政治敏感词库（含新华社禁用词）
│   ├── religious-sensitive.md          # 宗教敏感表述库
│   ├── terminology-standard.md         # 规范用语对照表
│   └── common-errors.md                # 常见错字语病例库
├── config.md                           # 审校配置
└── SKILL.md                            # 本文件
```

## 注意事项

1. **政治检查不可跳过**：无论稿件类型，政治敏感表述检查为必检项
2. **标注必须具体**：每个问题必须标注到段落位置，不能只说"某处有错"
3. **建议必须可操作**：不能只说"这里有问题"，必须给出修改方案
4. **存疑要诚实标注**：AI 不确定的信息标注为"存疑待确认"，不要猜测
5. **规则库持续积累**：每次审校发现新问题，都应更新对应规则库
6. **尊重原稿风格**：修改时不改变作者的写作风格，只修正客观错误
7. **格式检查适度**：小说、随笔等文学作品不过度干预格式选择
8. **新华社规范为核心依据**：`references/political-sensitive.md` 等规则库以《新华社新闻信息报道中的禁用词和慎用词》为基础，持续补充更新
